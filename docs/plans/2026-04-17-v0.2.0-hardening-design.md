# v0.2.0-hardening — Design

Date: 2026-04-17
Milestone: `v0.2.0-hardening`

## Goal

Harden the ColdCraft codebase against concurrency, security, and correctness issues surfaced in the 2026-04-17 codebase evaluation, and add the test/CI infrastructure to prevent regressions.

The app runs on a Claude Pro/Max subscription via `CLAUDE_CODE_OAUTH_TOKEN`, not on pay-per-token Anthropic API credits. All changes preserve that billing model.

## Scope

Seven discrete items, one GitHub issue per item, one PR per issue, all targeting `main`.

| # | Item | Type |
|---|------|------|
| 1 | Delete unused `src/lib/anthropic.ts` | Cleanup |
| 2 | Add Vitest + GitHub Actions CI skeleton | Infra |
| 3 | `PRO_TOKEN_SECRET` hard-fails in production + HMAC round-trip test | Security + test |
| 4 | Replace `execSync('claude -p')` with async `spawn` in `/api/generate` and `/api/scrape` | Reliability |
| 5 | Move `CREATE TABLE` DDL to `instrumentation.ts` + simplify rate-limit SQL + integration test | Correctness + test |
| 6 | Stripe webhook signature unit test | Test |
| 7 | Rewrite README with real setup, env, Docker, and subscription-model notes | Docs |

## Ordering

Execute in the order above. Rationale:

1. `anthropic.ts` delete — trivial, unblocks the rest mentally.
2. Vitest + CI skeleton — foundation; every later PR adds tests into it.
3. `PRO_TOKEN_SECRET` — security-critical; test lands with the subject.
4. `spawn` rewrite — largest behavioral change; lands after CI is live.
5. DDL + SQL cleanup — DB work grouped with its test.
6. Webhook test — standalone.
7. README — last, so it reflects the final state.

## Per-item design

### Item 1 — Delete `src/lib/anthropic.ts`

File is unused. `grep -r "from '@/lib/anthropic'"` returns nothing. Remove it; no other changes.

### Item 2 — Vitest + CI skeleton

- Add dev deps: `vitest`, `@vitest/ui`, `vite-tsconfig-paths`.
- Add scripts: `"test": "vitest run"`, `"test:watch": "vitest"`.
- `vitest.config.ts` — node environment, TS path resolution via `vite-tsconfig-paths`.
- `src/lib/__tests__/smoke.test.ts` — trivial green test so CI has something to run.
- `.github/workflows/test.yml`:
  - Triggers: `push`, `pull_request`.
  - Services: `postgres:16` on 5432 with health check (wired now; first consumer is item 5).
  - Steps: checkout → setup-node 20 with npm cache → `npm ci` → `npm run lint` → `npx tsc --noEmit` → `npm test`.
  - Env passed to test step: `DATABASE_URL`, `PRO_TOKEN_SECRET`, `STRIPE_WEBHOOK_SECRET` (test values).

### Item 3 — `PRO_TOKEN_SECRET` hard-fail in production

In `src/lib/pro-token.ts`, at module top level:

- If `process.env.PRO_TOKEN_SECRET` is missing or empty:
  - When `NODE_ENV === 'production'` → `throw new Error('PRO_TOKEN_SECRET is required in production')`.
  - Otherwise → `console.warn(...)` and fall back to `'fallback-dev-secret'`.

Test (`pro-token.test.ts`):

- Sign + verify round-trip with a known email.
- Tamper test: flip a byte in the HMAC, expect `verify → false`.
- Production-throw test: set `NODE_ENV=production`, unset `PRO_TOKEN_SECRET`, dynamic-import the module, expect throw.

### Item 4 — `execSync` → async `spawn`

New helper `src/lib/claude-cli.ts`:

```ts
runClaude(prompt: string, opts: { timeoutMs: number; maxBufferBytes?: number }): Promise<string>
```

Internals:

- `spawn('claude', ['-p', '-'])` with env = `{ ...process.env, ANTHROPIC_API_KEY: undefined, CLAUDE_CODE_OAUTH_TOKEN: process.env.CLAUDE_CODE_OAUTH_TOKEN ?? '' }`.
- Write `prompt` to stdin, `end()`.
- Accumulate stdout into a buffer; reject if total bytes exceed `maxBufferBytes`.
- Set a `setTimeout` for `timeoutMs` that calls `child.kill('SIGTERM')` (then `SIGKILL` after a grace period) and rejects.
- On `close`: resolve with stdout string if exit code === 0; reject with stderr otherwise.

Consumers:

- `src/app/api/generate/route.ts` — replace `execSync` block with `await runClaude(fullPrompt, { timeoutMs: 30_000, maxBufferBytes: 1_048_576 })`.
- `src/app/api/scrape/route.ts` — same, with `timeoutMs: 20_000, maxBufferBytes: 524_288`.

Output contract unchanged — both routes continue to parse JSON from the returned string.

Test (`claude-cli.test.ts`): mock `child_process.spawn` with a fake `EventEmitter` supporting `stdin.write`, `stdout.on('data')`, `stderr.on('data')`, `on('close')`. Cover four paths: happy, timeout, non-zero exit, buffer overflow.

### Item 5 — DDL startup move + rate-limit SQL simplification

- New `src/lib/schema.sql`:

  ```sql
  CREATE TABLE IF NOT EXISTS usage (
    fingerprint_id TEXT NOT NULL,
    usage_date DATE NOT NULL DEFAULT CURRENT_DATE,
    count INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (fingerprint_id, usage_date)
  );
  ```

- New `instrumentation.ts` at project root (Next 16 convention — runs once per server boot):

  ```ts
  export async function register() {
    if (process.env.NEXT_RUNTIME === 'nodejs') {
      const { initDb } = await import('./src/lib/db');
      await initDb().catch((err) => console.error('[instrumentation] initDb failed:', err));
    }
  }
  ```

- `src/lib/db.ts` changes:
  - New exported `initDb()` — reads `schema.sql`, runs against the pool.
  - `getDb()` — no longer runs DDL per request. Keeps a one-shot guard (`ddlRan` module-level flag) that runs `initDb()` on first call as a safety net in case `instrumentation.ts` didn't fire.
  - Documented in a short comment in `db.ts`.

- `src/app/api/generate/route.ts` rate-limit block becomes a single statement:

  ```sql
  INSERT INTO usage (fingerprint_id, usage_date, count)
  VALUES ($1, CURRENT_DATE, 1)
  ON CONFLICT (fingerprint_id, usage_date)
  DO UPDATE SET count = LEAST(usage.count + 1, $2)
  RETURNING count, (usage.count + 1 > $2) AS was_limited
  ```

  Parameters: `[fingerprint, DAILY_LIMIT]`. Drops the rollback UPDATE entirely.

Test (`generate-ratelimit.test.ts`, integration against Postgres service container):

- Before each: `TRUNCATE usage`.
- Hit the route handler 3 times → expect 200 each, `count` 1→2→3.
- 4th hit → expect 429, `was_limited === true`, count stays at 3.
- New-fingerprint path sets the cookie.

### Item 6 — Stripe webhook signature test

Test (`stripe-webhook.test.ts`):

- Build a test `Stripe.Event`; serialize; generate signed header via `stripe.webhooks.generateTestHeaderString({ payload, secret })`.
- POST to the route handler (direct function call, not HTTP).
- Positive: valid signature → 200 `{ received: true }`.
- Negative: wrong secret → 400.

## Testing strategy

### Unit tests (`src/lib/__tests__/`)

No external deps.

- `pro-token.test.ts` — HMAC round-trip, tamper, prod-throw.
- `claude-cli.test.ts` — mocked spawn, four paths.
- `stripe-webhook.test.ts` — signature positive/negative.

### Integration tests (`src/app/api/__tests__/`)

Need Postgres.

- `generate-ratelimit.test.ts` — rate-limit SQL behavior against real Postgres.

### Out of scope for v0.2.0

- E2E / browser tests.
- Snapshot tests on model output (non-deterministic).
- Coverage thresholds — add after suite stabilizes.

## CI

`.github/workflows/test.yml`:

- Triggers: `push`, `pull_request`.
- Job `test` on `ubuntu-latest`.
- Service: `postgres:16` on 5432, `POSTGRES_PASSWORD=postgres`, `pg_isready` health check.
- Steps: checkout → setup-node@v4 (20, npm cache) → `npm ci` → `npm run lint` → `npx tsc --noEmit` → `npm test`.
- Test env: `DATABASE_URL=postgres://postgres:postgres@localhost:5432/postgres`, `PRO_TOKEN_SECRET=test-secret-long-enough-for-hmac`, `STRIPE_WEBHOOK_SECRET=whsec_test`.

## Risks & mitigations

- **`spawn` regression (item 4).** Process plumbing is easy to get wrong. Mitigation: four-path test on `claude-cli`; land on a low-traffic window; watch logs.
- **DDL moving out of hot path (item 5).** First request after deploy could race `instrumentation.ts`. Mitigation: `getDb()` keeps a guarded one-shot `initDb()` as safety net.
- **Hard-failing on `PRO_TOKEN_SECRET` (item 3).** Could brick production if env var absent at deploy. Mitigation: re-verify `.env.production` contains it on the branch before merge.
- **CI Postgres flakiness (item 2, 5).** Mitigation: `pg_isready` health check; pin to `postgres:16`.

## Non-goals

- No change to fingerprint-cookie free-tier model.
- No `staging` branch or staged promotion flow.
- No extraction of pricing constants to a shared module.
- No Stripe customer portal / subscription management UI.

## Exit criteria

1. All 7 issues closed, 7 PRs merged to `main`.
2. `npm test` + `tsc --noEmit` + `eslint` green on `main`.
3. CI runs and passes on every PR.
4. Production `/api/generate` and `/api/scrape` verified post-item-4 deploy.
